# 自助台球厅通用设备控制模型设计说明

## 一、背景与目标

自助台球厅场景中，用户通过小程序扫码即可开台、支付、使用。
 在开台后，需要系统自动执行一系列「物理控制动作」，例如：

- 开启球桌上方灯光；
- 播放提示语音；
- 解除设备锁（球杆锁/桌锁）；
- 未来可能控制空调、电源等设备。

**目标：**
 设计一套通用的设备控制模型，使业务系统能够以统一的方式执行各种控制动作，而不依赖具体的硬件实现或 IoT 平台。

------

## 二、设计原则

1. **解耦业务与设备控制逻辑**
    开台、关台、支付等业务逻辑不应直接与设备控制代码耦合。
2. **协议可扩展、设备可配置**
    不同设备可使用不同协议（如 MQTT、HTTP 等），系统可通过配置灵活支持。
3. **统一抽象与标准化执行流程**
    设备信息、控制命令、执行引擎三者统一抽象，形成通用可扩展模型。

------

## 三、总体架构

### 1. 三层抽象模型

| 层级                        | 说明                                     | 示例                   |
| --------------------------- | ---------------------------------------- | ---------------------- |
| **设备模型（Device）**      | 定义设备基础信息（类型、协议、唯一标识） | 灯、音响、锁           |
| **控制命令模型（Command）** | 定义可执行的动作（开/关、播放、解锁）    | turn_on, play_audio    |
| **执行引擎（Executor）**    | 按设备协议发送命令                       | MQTT执行器、HTTP执行器 |

三者关系如下：

```
业务事件 → 控制命令集合 → 执行引擎选择 → 实际控制设备
```

------

### 2. 设备模型（Device）

设备是控制的最小单元。
 每个设备都应具备：

- 唯一标识（code）
- 类型（light / lock / speaker）
- 协议（mqtt / http / modbus）
- 协议参数配置（如 topic、ip、port）
- 状态（on/off 等）

设备可以通过后台手动配置，也可通过设备主动注册上报。

------

### 3. 控制命令模型（Command）

控制命令是业务与设备之间的“桥梁”，定义了系统希望设备执行的动作。
 例如：

| 场景     | 命令动作   | 说明         |
| -------- | ---------- | ------------ |
| 开台     | turn_on    | 打开台灯     |
| 关台     | turn_off   | 关闭台灯     |
| 支付完成 | play_audio | 播放语音提示 |

命令可以包含参数（如音量、时长、播放文件路径等）。

------

### 4. 执行引擎（Executor）

执行引擎根据设备协议，负责下发实际控制命令。

- **MQTT执行器**：向指定Topic发送控制指令。
- **HTTP执行器**：调用设备端API实现控制。
- **其他协议执行器**：如Modbus、TCP Socket等。

执行器通过工厂模式动态选择，实现多协议兼容。

------

## 四、业务层集成方式

### 1. 核心原则

**设备控制由后端业务事件自动触发，前端无需感知设备控制细节。**

用户通过小程序扫码开台后，后端自动完成：
- 创建订单
- 触发设备控制（开灯、解锁等）
- 返回开台成功

前端只需关注业务结果（开台成功/失败），设备控制完全由后端处理。

------

### 2. 业务触发点与控制动作映射

| 业务事件       | 触发时机           | 自动执行的设备控制动作         |
| -------------- | ------------------ | ------------------------------ |
| **开台**       | 用户扫码开台成功   | 开灯、播放欢迎语音、解锁球杆锁 |
| **关台/结算**  | 订单结算完成       | 关灯、锁定球杆锁               |
| **台费充值**   | 充值到账           | 播放到账提示语音               |
| **超时提醒**   | 订单即将到期       | 播放超时提醒语音               |
| **强制关台**   | 管理员远程关台     | 关灯、锁定设备                 |
| **设备维护**   | 后台设备管理操作   | 重启设备、测试设备             |

------

### 3. 后端处理流程

以**开台**为例：

```
用户扫码 → Controller 接收请求
         ↓
    Service 层处理业务逻辑：
         ↓
    1. 校验台桌状态（是否可用）
         ↓
    2. 创建订单（状态：使用中）
         ↓
    3. 调用 IoT 控制层（异步）
         ↓
         ├─ 查询设备绑定配置
         │  SELECT * FROM iot_device_binding 
         │  WHERE table_id = ? AND scene = 'open_table' 
         │  AND enabled = 1 
         │  ORDER BY execute_order
         ↓
         ├─ 按顺序执行设备控制
         │  ├─ LIGHT-001: turn_on (顺序1)
         │  ├─ LOCK-001: unlock (顺序2)
         │  └─ SPEAKER-001: play_audio (顺序3)
         ↓
         ├─ 通过执行引擎下发指令
         │  └─ 根据设备协议选择执行器（MQTT/HTTP）
         ↓
         └─ 记录控制日志（iot_control_log）
             └─ 失败时触发告警（iot_device_alert）
         ↓
    4. 返回前端：开台成功
```

**关键设计点：**
- **配置驱动**：通过 `iot_device_binding` 表查询需要执行的设备动作，而不是硬编码
- **异步非阻塞**：设备控制不阻塞业务主流程，即使设备响应慢也不影响开台成功
- **顺序控制**：按 `execute_order` 字段顺序执行（如先开灯再播放语音）
- **最终一致性**：业务状态优先更新，设备状态稍后同步
- **操作日志**：所有控制动作都记录日志，失败时自动产生告警

------

## 五、数据模型设计

### 1. 数据模型关系图

```
┌─────────────────┐
│  bls_table     │  台桌表（业务表）
│  - id           │
│  - table_no     │
│  - store_id     │
└────────┬────────┘
         │
         │ 1:N
         ↓
┌─────────────────────────┐      N:1      ┌─────────────────┐
│  iot_device_binding     │───────────────→│  iot_device     │  设备表
│  - id                   │                │  - id           │
│  - table_id (FK)        │                │  - code (UK)    │
│  - device_code (FK)     │                │  - name         │
│  - scene                │                │  - type         │
│  - command              │                │  - protocol     │
│  - execute_order        │                │  - status       │
└────────┬────────────────┘                └────────┬────────┘
         │                                          │
         │ 触发执行                                 │ 产生日志/告警
         ↓                                          ↓
┌─────────────────────────┐                ┌─────────────────────┐
│  iot_control_log        │                │  iot_device_alert   │
│  - id                   │                │  - id               │
│  - device_code (FK)     │                │  - device_code (FK) │
│  - table_id             │                │  - alert_type       │
│  - scene                │                │  - alert_level      │
│  - command              │                │  - status           │
│  - status               │                │  - handler          │
│  - execute_time         │                └─────────────────────┘
└─────────────────────────┘
```

**核心关系说明：**

- `bls_table` (1) ←→ (N) `iot_device_binding`：一个台桌可以配置多个设备控制规则
- `iot_device` (1) ←→ (N) `iot_device_binding`：一个设备可以被多个台桌/场景使用
- `iot_device_binding` → `iot_control_log`：每次执行产生一条控制日志
- `iot_device` → `iot_device_alert`：设备异常时产生告警记录

------

### 2. 核心表结构

#### 设备表（iot_device）

| 字段             | 类型         | 说明                               |
| ---------------- | ------------ | ---------------------------------- |
| id               | bigint       | 主键                               |
| code             | varchar(100) | 设备唯一编号（唯一索引）           |
| name             | varchar(200) | 设备名称                           |
| type             | varchar(50)  | 设备类型：light/lock/speaker/other |
| protocol         | varchar(50)  | 协议类型：mqtt/http/modbus         |
| protocol_config  | json         | 协议配置（topic/ip/port等）        |
| status           | varchar(20)  | 设备状态：online/offline/error     |
| store_id         | bigint       | 所属门店ID                         |
| last_heartbeat   | datetime     | 最后心跳时间                       |
| remark           | varchar(500) | 备注信息                           |
| create_time      | datetime     | 创建时间                           |
| update_time      | datetime     | 更新时间                           |

**说明**：设备表只存储设备本身的信息，不直接绑定台桌。设备与台桌的关系通过 `iot_device_binding` 表维护。

#### 设备控制日志表（iot_control_log）

| 字段         | 类型         | 说明                             |
| ------------ | ------------ | -------------------------------- |
| id           | bigint       | 主键                             |
| device_code  | varchar(100) | 设备编号                         |
| command      | varchar(50)  | 控制命令：turn_on/turn_off/play  |
| params       | json         | 命令参数                         |
| trigger_by   | varchar(50)  | 触发来源：order/admin/system     |
| trigger_scene | varchar(50) | 触发场景：open_table/close_table |
| order_id     | bigint       | 关联订单ID                       |
| table_id     | bigint       | 关联台桌ID                       |
| status       | varchar(20)  | 执行状态：success/failed/timeout |
| error_msg    | text         | 失败原因                         |
| retry_count  | int          | 重试次数                         |
| execute_time | datetime     | 执行时间                         |
| response_time | int         | 响应耗时（毫秒）                 |

#### 设备业务绑定表（iot_device_binding）

**核心作用：定义业务场景与设备控制动作的映射关系**

| 字段         | 类型         | 说明                                      |
| ------------ | ------------ | ----------------------------------------- |
| id           | bigint       | 主键                                      |
| table_id     | bigint       | 台桌ID（外键）                            |
| store_id     | bigint       | 门店ID                                    |
| scene        | varchar(50)  | 业务场景：open_table/close_table/timeout  |
| device_code  | varchar(100) | 设备编号（外键）                          |
| command      | varchar(50)  | 控制命令：turn_on/turn_off/play_audio     |
| params       | json         | 命令参数（如音量、播放文件等）            |
| execute_order | int         | 执行顺序（同一场景多个设备时的执行顺序）  |
| enabled      | tinyint      | 是否启用：1-启用，0-禁用                  |
| create_time  | datetime     | 创建时间                                  |

**示例数据：**

```sql
-- 1号台开台时的设备控制配置
INSERT INTO iot_device_binding VALUES 
(1, 1, 1, 'open_table', 'LIGHT-001', 'turn_on', NULL, 1, 1, NOW()),
(2, 1, 1, 'open_table', 'LOCK-001', 'unlock', NULL, 2, 1, NOW()),
(3, 1, 1, 'open_table', 'SPEAKER-001', 'play_audio', '{"file":"welcome.mp3","volume":80}', 3, 1, NOW());

-- 1号台关台时的设备控制配置
INSERT INTO iot_device_binding VALUES 
(4, 1, 1, 'close_table', 'LIGHT-001', 'turn_off', NULL, 1, 1, NOW()),
(5, 1, 1, 'close_table', 'LOCK-001', 'lock', NULL, 2, 1, NOW());
```

**设计优势：**
- ✅ **配置化**：业务人员可通过后台界面配置，无需改代码
- ✅ **灵活性**：不同台桌可配置不同的设备组合
- ✅ **可扩展**：新增业务场景（如超时提醒）只需新增配置
- ✅ **顺序控制**：通过 execute_order 控制设备执行顺序（如先开灯再解锁）
- ✅ **动态开关**：可临时禁用某个设备控制而不删除配置

#### 设备告警表（iot_device_alert）

| 字段          | 类型         | 说明                                        |
| ------------- | ------------ | ------------------------------------------- |
| id            | bigint       | 主键                                        |
| alert_type    | varchar(50)  | 告警类型：offline/control_failed/low_success_rate |
| alert_level   | varchar(20)  | 告警级别：info/warning/error/critical       |
| device_code   | varchar(100) | 设备编号                                    |
| device_name   | varchar(200) | 设备名称                                    |
| table_id      | bigint       | 关联台桌ID                                  |
| store_id      | bigint       | 所属门店ID                                  |
| alert_content | text         | 告警内容描述                                |
| alert_data    | json         | 告警详细数据（如失败次数、成功率等）        |
| status        | varchar(20)  | 处理状态：pending/processing/resolved/ignored |
| handler       | varchar(100) | 处理人                                      |
| handle_time   | datetime     | 处理时间                                    |
| handle_remark | text         | 处理备注                                    |
| create_time   | datetime     | 告警产生时间                                |
| resolved_time | datetime     | 解决时间                                    |

**告警类型说明：**

| 告警类型          | 触发条件                   | 告警级别 |
| ----------------- | -------------------------- | -------- |
| offline           | 5分钟未收到心跳            | error    |
| control_failed    | 控制指令连续失败3次        | warning  |
| low_success_rate  | 1小时内成功率 < 80%        | warning  |
| device_error      | 设备主动上报错误状态       | error    |
| batch_offline     | 同门店50%设备离线          | critical |
| timeout_frequent  | 频繁超时（10次/小时）      | warning  |

### 2. 设备注册与配置

#### 手动注册（推荐起步方式）

管理员在后台录入设备：
1. 填写设备基本信息（编号、名称、类型）
2. 选择协议类型（MQTT/HTTP）
3. 配置协议参数：
   - MQTT：`{"broker": "tcp://localhost:1883", "topic": "billiards/table/1/light"}`
   - HTTP：`{"url": "http://192.168.1.55:8080/control", "method": "POST"}`
4. 绑定到指定台桌

#### 自动注册（高级扩展）（采用该方式）

设备上电后主动上报注册信息：

```json
{
  "code": "LIGHT-001",
  "type": "light",
  "protocol": "mqtt",
  "config": {
    "topic": "billiards/light/1"
  },
  "mac": "AA:BB:CC:DD:EE:FF"
}
```

系统接收后自动创建或更新设备记录。

## 六、异常处理与容错机制

### 1. 设备控制失败场景

| 异常类型       | 可能原因                   | 处理策略                                   |
| -------------- | -------------------------- | ------------------------------------------ |
| **设备离线**   | 断电、网络断开             | 记录日志，标记设备状态为 offline，告警通知 |
| **指令超时**   | 设备响应慢、网络延迟       | 默认 5 秒超时，超时后记录失败日志          |
| **指令执行失败** | 设备故障、协议错误       | 记录错误信息，不影响业务主流程             |
| **协议配置错误** | Topic 错误、IP 不可达    | 后台配置校验，执行前验证连通性             |

------

### 2. 容错原则

**核心思想：设备控制失败不影响核心业务流程**

- **业务优先**：开台、结算等核心操作不因设备问题而失败
- **异步处理**：设备控制采用异步方式，不阻塞主流程
- **日志记录**：所有控制动作都记录日志，便于排查
- **人工兜底**：设备异常时，店员可手动处理物理设备

```

- 最多重试 3 次
- 重试间隔：1 秒、2 秒、4 秒（指数退避）
- 仅针对可重试的异常（如超时、连接失败）

------

## 七、监控与告警

### 1. 设备健康监控

- **心跳检测**：设备定期上报心跳（每 30 秒一次）
- **状态监控**：后台定时检查设备在线状态
- **控制成功率**：统计设备控制的成功率，低于 80% 时告警

------

### 2. 告警产生与处理流程

#### 告警产生规则

| 告警类型             | 触发条件                | 告警级别 | 通知方式          |
| -------------------- | ----------------------- | -------- | ----------------- |
| **设备离线**         | 5 分钟未收到心跳        | error    | 管理后台 + 短信   |
| **控制失败率高**     | 1 小时内失败率 > 20%    | warning  | 管理后台          |
| **批量设备异常**     | 同一门店 50% 设备离线   | critical | 短信 + 电话       |
| **关键设备故障**     | 灯光设备连续失败 3 次   | error    | 实时告警          |
| **设备响应慢**       | 平均响应时间 > 3 秒     | info     | 管理后台          |

#### 告警处理流程

```
设备异常 → 产生告警记录
         ↓
    INSERT INTO iot_device_alert
    (alert_type, alert_level, device_code, alert_content, status)
         ↓
    判断告警级别：
         ↓
    ├─ critical → 立即短信+电话通知
    ├─ error    → 管理后台+短信通知
    ├─ warning  → 管理后台通知
    └─ info     → 仅记录日志
         ↓
    运维人员处理：
         ↓
    UPDATE iot_device_alert 
    SET status = 'processing', handler = '张三'
         ↓
    处理完成后：
         ↓
    UPDATE iot_device_alert 
    SET status = 'resolved', handle_remark = '已更换设备'
```

#### 告警数据示例

```sql
-- 设备离线告警
INSERT INTO iot_device_alert (
    alert_type, alert_level, device_code, device_name, 
    table_id, store_id, alert_content, alert_data, 
    status, create_time
) VALUES (
    'offline', 'error', 'LIGHT-001', '1号台灯光', 
    1, 1, '设备已离线超过5分钟', 
    '{"last_heartbeat":"2025-01-15 10:00:00","offline_duration":300}',
    'pending', NOW()
);

-- 控制失败告警

## 九、方案优势

- **业务无感知**：设备控制完全由后端自动化处理，前端无需关心设备细节
- **通用性强**：支持任意类型设备，易于扩展新设备类型
- **协议兼容**：通过执行器模式支持多种通信协议
- **容错可靠**：设备异常不影响核心业务，保障系统稳定性
- **易于维护**：统一的抽象模型，降低开发和运维复杂度
- **部署灵活**：可独立模块化部署或集成至现有系统

## 十一、总结

本方案通过**"设备抽象 + 协议分发 + 命令驱动 + 后端自动化"**的设计思路，实现了：

1. **通用可扩展的设备控制层**：统一抽象模型，支持任意设备类型和协议
2. **对上层业务的无感集成**：业务事件自动触发设备控制，前端无需感知
3. **对下层硬件的协议兼容**：通过执行器工厂模式支持多种通信协议
4. **完善的容错机制**：设备异常不影响核心业务，保障系统可用性

该模型可作为**自助台球厅、无人值守门店、智能场馆**等业务的基础控制框架，支持持续扩展与演进。
