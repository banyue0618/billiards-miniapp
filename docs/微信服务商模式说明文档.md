# 微信服务商模式说明文档

## 一、整体支付逻辑

### 1. 平台角色

- 平台申请一个微信支付服务商商户号（mchId）
- 入驻商家称为“子商户”，每个商家申请一个 subMchId

### 2. 支付流程

1. 用户在平台小程序选择某个门店/商家
2. 平台后端读取该门店对应的 subMchId
3. 使用平台自己的 mchId、API密钥、证书 进行统一下单
4. 在请求参数中传入 subMchId 表示该订单归属的商户
5. 用户支付，微信将资金结算到该子商户账户
6. 平台可以通过微信分账接口收取佣金

## 二、参数配置说明

| 参数名称          | 平台是否固定配置 | 商家是否独立提供  |
| ----------------- | ---------------- | ----------------- |
| mchId             | ✅                | ❌                 |
| apiKey / apiV3Key | ✅                | ❌                 |
| 证书 cert/keyPath | ✅                | ❌                 |
| subMchId          | ❌                | ✅（每个商家一个） |
| subAppId（可选）  | ❌                | ✅（部分情况）     |

## 三、常见疑问解答

### Q1：每个商家是不是要配置一套证书、密钥？

**不是。**只有平台服务商需要准备一套支付证书和密钥。每个商家只要提供 subMchId。

### Q2：支付接口参数这么多，我是不是要为每个商家复制好多配置？

**不需要。**所有支付请求仍使用平台自己的配置，只是多传一个 subMchId。

### Q3：这样会不会很复杂，后台要不要重写很多代码？

**只要在下单前查一下门店绑定的 subMchId，动态赋值即可，其他逻辑和统一支付基本一致。**

### Q4：如果商家不想自己去申请商户号怎么办？

那只能走平台统一收款再转账的模式，但这种属于二清模式，合规风险高。

## 四、平台的优势

- 商家不需要理解复杂的接口，只提供 subMchId
- 支付合规，不触碰资金
- 可通过微信分账接口自动收取佣金

## 五、问题推导过程回顾

- 起初疑问：用户在平台小程序支付，应该是直接付给商家还是先到平台再转？是否属于二清？
- 讨论结论：平台如果统一收款再转账属于二清风险较高，推荐使用微信服务商模式，资金直接到商家。
- 接着疑问：服务商模式下是否需要为每个商家配置一整套支付证书、密钥？
- 解答：不需要，平台只需配置一套自己的商户密钥与证书，商家只需要提供 subMchId。
- 最后疑问：支付接口会不会很复杂？
- 解答：只需要在支付参数中动态传入 subMchId，其他所有配置仍是平台自己的，技术上只是查库+赋值即可。
