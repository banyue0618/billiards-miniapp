## 1. 系统概述

### 1.1 系统目标

自助台球厅系统旨在提供一个完整的解决方案，帮助台球厅实现无人化或少人化运营，用户可通过微信小程序完成查找门店、扫码开台、自助计时计费等操作，商家可通过后台系统进行门店管理、桌台管理、订单查看等操作。

### 1.2 系统架构

#### 1.2.1 整体架构
自助台球厅系统采用前后端分离的架构设计，通过RESTful API实现前后端通信。系统架构分为三层：

1. **接入层**：负责处理用户请求，包括小程序前端、商家管理后台、负载均衡等
2. **应用层**：核心业务逻辑实现，包括用户服务、门店服务、桌台服务、订单服务等
3. **数据层**：数据存储和管理，包括关系型数据库、缓存系统、对象存储等

#### 1.2.2 初期架构（单机部署）
初创阶段采用单机部署模式，适合小规模用户量和低并发场景：

```
                    +----------------+
                    |  负载均衡(可选) |
                    +--------+-------+
                             |
+----------------+  +--------v-------+  +----------------+
| 微信小程序前端  +--+   应用服务器    +--+  管理后台前端   |
+----------------+  | (All-in-One)  |  +----------------+
                    +--------+-------+
                             |
                    +--------v-------+
                    |    MySQL数据库  |
                    +----------------+
```

- 单台服务器部署全部后端服务
- 单一数据库实例
- 简单的文件系统存储静态资源
- 基础的监控和告警机制

#### 1.2.3 中期架构（垂直拆分）
随着业务增长，将系统垂直拆分为多个独立服务：

```
                    +----------------+
                    |   API网关      |
                    +--------+-------+
                             |
+----------------+           |           +----------------+
| 微信小程序前端  |           |           |  管理后台前端   |
+-------+--------+           |           +-------+--------+
        |                    |                   |
        +-------+    +-------v--------+   +-----+
                |    |                |   |
        +-------v----v---+   +--------v---v----+   +---------------+
        |  用户服务      |   |   订单服务      |   |  门店管理服务  |
        +-------+--------+   +--------+-------+   +--------+------+
                |                     |                    |
        +-------v--------+   +--------v-------+   +-------v-------+
        |  用户数据库     |   |   订单数据库    |   |  门店数据库   |
        +----------------+   +----------------+   +---------------+
                                     |
                            +--------v-------+
                            |   Redis缓存    |
                            +----------------+
```

- 按业务功能垂直拆分为多个服务
- 独立数据库和缓存系统
- 引入API网关统一接口管理
- 加入消息队列处理异步任务
- 实现服务注册与发现

#### 1.2.4 长期架构（微服务化）
业务规模进一步扩大，转向完整的微服务架构：

- 完整的微服务架构，服务粒度更细

- CI/CD自动化部署流水线

- **开发语言和框架**
  - 核心语言：Java

  - 关系型数据库：MySQL

  - 缓存系统：Redis

- **通信与集成**
  - API规范：RESTful
  - API文档：Swagger / OpenAPI
  - 消息队列：Kafka / RabbitMQ

#### 1.3.3 DevOps工具链

- **构建与部署**
  - 容器化：Docker
  - 容器编排：Kubernetes

  - 配置管理：Spring Cloud Config / Nacos

- **监控与运维**
  - APM：Prometheus + Grafana
  - 日志聚合：ELK Stack (Elasticsearch, Logstash, Kibana)

#### 1.3.4 安全与认证

- **安全框架**：JWT，框架采用sa-token
- **加密算法**：HTTPS / SHA-256 / AES
- **数据脱敏**：字段级加密与脱敏

- **权限控制**：RBAC (基于角色的访问控制)

  - CDN加速：阿里云CDN / 腾讯云CDN

#### 1.3.6 演进策略

  - 服务完全微服务化
  - 实现容器化编排与自动扩缩容
  - 多区域部署和就近接入
  - 完善的监控、告警与自愈系统

#### 1.3.7 静态资源存储策略

静态资源（图片、视频等）的存储是系统设计中重要的一环，为了兼顾前期开发的简便性和后期扩展的灵活性，我们采用渐进式的资源存储策略，通过配置切换机制实现无缝升级。

##### 1.3.7.1 存储方案演进

**阶段一：本地文件系统存储（初期）**

适用条件：
- 门店数量少（50家以内）
- 用户量小（日活1000以内）
- 静态资源总量小于5GB
- 单机或小规模集群部署

实现方式：
- 使用服务器本地文件系统直接存储图片
- 设计合理的目录结构，例如按资源类型和业务ID组织：
  ```
  /data/images/
    ├── stores/            # 门店相关图片
    │   ├── store_1/       # 按门店ID组织
    │   │   ├── cover.jpg  # 门店封面图
    │   │   └── gallery/   # 门店环境图集
    ├── tables/            # 桌台相关图片
    │   ├── table_1.jpg
    │   └── ...
    └── qrcodes/           # 二维码图片
        ├── table_1.png
        └── ...
  ```
- 使用Spring MVC的静态资源映射提供图片访问服务

优势：
- 实现简单，开发调试方便
- 无需额外的组件和成本
- 小规模业务场景下性能足够

**阶段二：对象存储服务（中期）**

适用条件：
- 门店数量增长（100+家）
- 日活增长到5000+
- 静态资源总量超过10GB
- 需要更可靠的备份和异地容灾能力

实现方式：
- 选择方案一：云服务商提供的对象存储（阿里云OSS、腾讯云COS等）
- 选择方案二：自建MinIO对象存储服务

配置参数：
- 存储桶（Bucket）配置
- 访问密钥配置
- 基础URL配置
- 权限策略配置

优势：
- 提供更高的可用性和数据安全性
- 支持对象生命周期管理
- 支持访问控制和权限管理
- 可配置防盗链和私有资源访问控制

**阶段三：CDN加速分发（长期）**

适用条件：
- 业务扩展至多城市/地区
- 日活用户超过1万
- 对资源加载速度有较高要求
- 系统整体进入分布式架构阶段

实现方式：
- 对象存储 + CDN服务结合使用
- 对象存储作为CDN源站
- CDN服务提供全国/全球加速节点

配置参数：
- CDN域名配置
- 缓存策略配置
- HTTPS配置
- 回源规则配置

优势：
- 显著提升资源加载速度
- 减轻源站压力
- 支持HTTPS安全访问
- 提供DDoS防护能力

##### 1.3.7.2 配置切换机制设计

为了实现不同存储方案的平滑切换，我们采用策略模式设计资源存储服务：

**1. 资源服务接口定义**

```java
/**
 * 资源服务接口
 * 提供统一的资源上传、获取URL等操作
 */
public interface ResourceService {
    /**
     * 上传资源
     * @param inputStream 资源输入流
     * @param fileName 文件名
     * @param resourceType 资源类型（枚举）
     * @return 资源ID
     */
    String uploadResource(InputStream inputStream, String fileName, ResourceType resourceType);
    
    /**
     * 获取资源访问URL
     * @param resourceId 资源ID
     * @param resourceType 资源类型
     * @return 完整的访问URL
     */
    String getResourceUrl(String resourceId, ResourceType resourceType);
    
    /**
     * 删除资源
     * @param resourceId 资源ID
     * @param resourceType 资源类型
     * @return 是否删除成功
     */
    boolean deleteResource(String resourceId, ResourceType resourceType);
    
    /**
     * 生成私有资源的临时访问URL
     * @param resourceId 资源ID
     * @param resourceType 资源类型
     * @param expireTime 过期时间（分钟）
     * @return 带签名的临时访问URL
     */
    String generatePresignedUrl(String resourceId, ResourceType resourceType, int expireTime);
}
```

**2. 配置定义**

在`application.properties`或`application.yml`中定义：

```yaml
# 资源存储配置
resource:
  # 存储类型：local(本地文件系统), minio(MinIO对象存储), oss(云服务商对象存储), cdn(CDN加速)

  # 本地存储配置
  local:
    base-path: /data/images
    base-url: http://api.example.com/images
  
  # MinIO配置
  minio:
    endpoint: http://minio.example.com:9000
    access-key: minioadmin
    secret-key: minioadmin
    bucket: billiards-app
    base-url: http://minio.example.com:9000/billiards-app
  
  # 阿里云OSS配置
  oss:
    endpoint: https://oss-cn-hangzhou.aliyuncs.com
    access-key-id: YOUR_ACCESS_KEY
    access-key-secret: YOUR_SECRET_KEY
    bucket: billiards-app
    base-url: https://billiards-app.oss-cn-hangzhou.aliyuncs.com
  
  # CDN配置
  cdn:
    # CDN仍然需要一个存储后端，这里使用oss
    storage-backend: oss
    domain: https://static.example.com
    private-key: YOUR_CDN_PRIVATE_KEY
```

**4. 服务工厂**

```java
@Service
public class ResourceServiceFactory {
    @Autowired(required = false)
    @Qualifier("localResourceService")
    private ResourceService localResourceService;
    
    @Autowired(required = false)
    @Qualifier("minioResourceService")
    private ResourceService minioResourceService;
    
    @Autowired(required = false)
    @Qualifier("ossResourceService")
    private ResourceService ossResourceService;
    
    @Autowired(required = false)
    @Qualifier("cdnResourceService")
    private ResourceService cdnResourceService;
    
    @Value("${resource.storage-type}")
    private String storageType;
    
    public ResourceService getResourceService() {
        switch (storageType) {
            case "local":
                return localResourceService;
            case "minio":
                return minioResourceService;
            case "oss":
                return ossResourceService;
            case "cdn":
                return cdnResourceService;
            default:
                throw new IllegalArgumentException("Unsupported storage type: " + storageType);
        }
    }
}
```

##### 1.3.7.3 使用示例

**业务服务中使用资源服务**

```java
@Service
public class StoreServiceImpl implements StoreService {
    @Autowired
    private ResourceServiceFactory resourceServiceFactory;
    
    @Autowired
    private StoreRepository storeRepository;
    
    @Override
    public void updateStoreCover(String storeId, MultipartFile coverImage) throws IOException {
        // 获取当前配置的资源服务
        ResourceService resourceService = resourceServiceFactory.getResourceService();
        
        // 上传图片
        String resourceId = resourceService.uploadResource(
                coverImage.getInputStream(), 
                coverImage.getOriginalFilename(),
                ResourceType.STORE_COVER
        );
        
        // 更新门店封面图URL
        Store store = storeRepository.findById(storeId)
                .orElseThrow(() -> new EntityNotFoundException("门店不存在"));
        
        // 保存资源ID而非完整URL，URL在访问时动态生成
        store.setCoverImageId(resourceId);
        storeRepository.save(store);
    }
    
    @Override
    public StoreDTO getStoreDetail(String storeId) {
        Store store = storeRepository.findById(storeId)
                .orElseThrow(() -> new EntityNotFoundException("门店不存在"));
        
        // 获取当前配置的资源服务
        ResourceService resourceService = resourceServiceFactory.getResourceService();
        
        // 构建DTO，动态生成资源URL
        StoreDTO dto = new StoreDTO();
        // 设置基本属性...
        
        // 动态生成封面图URL
        if (store.getCoverImageId() != null) {
            dto.setCoverImageUrl(
                    resourceService.getResourceUrl(store.getCoverImageId(), ResourceType.STORE_COVER)
            );
        }
        
        // 其他处理...
        return dto;
    }
}
```

##### 1.3.7.4 数据迁移

在存储方案升级过程中，需要考虑历史数据的迁移：

**1. 数据迁移工具**

创建单独的数据迁移工具，支持：
- 从本地文件系统迁移到MinIO/OSS
- 批量处理，支持断点续传
- 迁移过程记录与校验
- 迁移后的回滚机制

**2. 迁移步骤**

1. 准备新存储环境（MinIO/OSS）
2. 运行迁移工具，批量复制资源
3. 更新配置，切换存储类型
4. 验证新系统访问
5. 保留原存储一段时间作为备份

**3. 平滑过渡策略**

在迁移期间，可采用双写策略确保数据一致性：
- 新上传的资源同时写入旧存储和新存储
- 读取时优先尝试从新存储读取，失败则回退到旧存储
- 通过监控确认所有数据访问稳定后，完全切换到新存储

通过以上设计，系统可以根据业务发展需要，灵活地在不同存储方案间切换，同时保持代码的一致性和可维护性。

## 3. 扫码开台模块

扫码开台模块是用户进入球厅后，通过扫描球桌上的二维码快速完成开台并开始计费的核心功能模块。该模块实现了识别二维码、验证桌台状态、用户确认开台并触发时间计费等关键业务逻辑。

### 3.1 二维码生成规则

#### 3.1.1 二维码内容设计
- 每张桌台生成唯一二维码，包含以下信息：
  - 门店ID（storeId）：唯一标识门店
  - 桌台ID（tableId）：唯一标识桌台
  - 桌台类型（tableType）：标识桌台级别
  - 校验码（checksum）：防止篡改
- 使用短链接形式生成二维码，避免复杂参数暴露
- 二维码链接格式：`https://example.com/t/{加密参数}`

#### 3.1.2 二维码生成流程
1. 商家后台添加新桌台时自动生成
2. 系统根据桌台信息生成参数字符串
3. 参数字符串进行加密处理（如Base64编码+MD5校验）
4. 生成短链接URL
5. 基于短链接生成二维码图像
6. 提供下载和打印功能

#### 3.1.3 二维码安全性设计
- 链接使用HTTPS协议确保传输安全
- 所有参数均进行加密，防止直接篡改
- 服务端进行二维码有效性验证
- 支持二维码失效与更新机制

#### 3.1.4 技术实现方案
- 使用微信小程序码接口生成永久小程序码
- 二维码参数使用Base64编码并添加签名
- 服务端存储二维码映射关系，确保唯一性
- 二维码图片分辨率：420×420像素

### 3.2 扫码流程设计

#### 3.2.1 扫码入口
- **首页入口**：
  - 点击门店卡片上的【扫码开台】按钮
  - 调起微信扫码功能
- **扫一扫直达**：
  - 用户可直接使用微信"扫一扫"功能
  - 识别台球桌上的二维码
  - 自动跳转小程序相应页面
- **历史记录入口**：
  - 提供最近使用记录快速开台入口（可选）

#### 3.2.2 扫码完整流程
1. 用户点击【扫码开台】或使用微信扫一扫
2. 调用相机扫描二维码
3. 小程序接收并解析二维码信息
4. 向服务端发送请求验证二维码有效性
5. 服务端返回桌台详细信息（状态、计费规则等）

## 4. 计价模块（后端逻辑）

计价模块是商家设置和管理不同球桌收费方式的核心功能，支持灵活配置多种计费策略，实现自动化的费用计算。计价模块提供标准计费和阶梯计费两种基本模式，并可针对每张桌台设置独立的计费规则。

### 4.1 标准计费模式

#### 4.1.1 功能描述
标准计费是最基础的计费方式，使用固定的单价进行计费，适用于大多数简单计费场景。

#### 4.1.2 配置参数
- **基本参数**：
  - 计费单位：分钟或小时（系统内部统一以分钟为单位计算）
  - 单价：如 0.5 元/分钟 或 30 元/小时
  - 计费精度：精确到分钟或秒（默认分钟）
- **扩展参数**：
  - 最低消费时长：设置使用的最低计费时长（如至少30分钟）
  - 封顶价格：设置单次使用的最高费用上限（如最多80元）
  - 会员价格：设置会员用户享受的优惠单价（如会员0.4元/分钟）

#### 4.1.3 计费公式
```
// 基本计费公式
费用 = 使用时长(分钟) × 单价(元/分钟)

// 考虑最低消费
if (使用时长 < 最低消费时长) {
    使用时长 = 最低消费时长
}

// 考虑封顶价格
if (费用 > 封顶价格) {
    费用 = 封顶价格
}

// 会员价格计算
if (是会员) {
    费用 = 使用时长(分钟) × 会员单价(元/分钟)
    // 同样考虑最低消费和封顶价格
}
```

#### 4.1.4 使用示例
```
配置示例：
- 单价：0.5元/分钟
- 最低消费：30分钟
- 封顶价格：80元
- 会员单价：0.4元/分钟

计费示例1：普通用户使用25分钟
- 由于低于最低消费时长，按30分钟计算
- 费用 = 30分钟 × 0.5元/分钟 = 15元

计费示例2：会员用户使用180分钟
- 费用 = 180分钟 × 0.4元/分钟 = 72元
- 未超过封顶价格，所以费用为72元

计费示例3：普通用户使用200分钟
- 费用 = 200分钟 × 0.5元/分钟 = 100元
- 超过封顶价格，所以最终费用为80元
```

### 4.2 阶梯计费模式

#### 4.2.1 功能描述
阶梯计费是一种高级计费方式，可根据使用时长分段设置不同单价，通常用于实现"使用越久越便宜"的营销策略，鼓励用户长时间使用。

#### 4.2.2 配置参数
- **阶梯设置**：
  - 支持配置多个时间区间及对应单价
  - 每个区间需设置起始时间和结束时间（最后一段可设为无限）
  - 每个区间设置独立的单价
- **扩展参数**：
  - 最低消费设置（可选）
  - 会员阶梯价格（可设置独立的会员阶梯）
  - 封顶价格（可选）

#### 4.2.3 计费公式
```
// 阶梯计费核心逻辑
总费用 = 0
对于每个时间区间 i:
    if (使用时长 > 区间i起始时间) {
        区间内时长 = min(使用时长, 区间i结束时间) - 区间i起始时间
        区间费用 = 区间内时长 × 区间i单价
        总费用 += 区间费用
    }

// 考虑最低消费和封顶价格
if (总费用 < 最低消费) {
    总费用 = 最低消费
}
if (总费用 > 封顶价格 && 封顶价格 > 0) {
    总费用 = 封顶价格
}
```

#### 4.2.4 使用示例
```
配置示例：
- 阶梯1：0-60分钟，0.5元/分钟
- 阶梯2：61-120分钟，0.4元/分钟
- 阶梯3：121分钟以上，0.3元/分钟

计费示例：用户使用150分钟
- 阶梯1：60分钟 × 0.5元/分钟 = 30元
- 阶梯2：60分钟 × 0.4元/分钟 = 24元
- 阶梯3：30分钟 × 0.3元/分钟 = 9元
- 总费用 = 30 + 24 + 9 = 63元
```

#### 4.2.5 阶梯设置界面设计
- 动态添加/删除阶梯区间
- 阶梯区间之间自动首尾衔接（前一阶段结束时间即为下一阶段开始时间）
- 支持拖拽调整阶梯范围
- 可视化展示阶梯价格曲线图

### 4.3 最低消费设置

#### 4.3.1 功能描述
最低消费设置允许商家为每张球桌设置最低消费金额或时长，确保即使用户使用时间很短，也能达到最低营收要求。

#### 4.3.2 设置选项
- **按时长设置**：
  - 设置最低使用时长（如最少30分钟）
  - 低于此时长按最低时长计费
- **按金额设置**：
  - 设置最低消费金额（如最低20元）
  - 若实际计费金额低于设定值，则按最低金额收费

#### 4.3.3 应用逻辑
- 最低消费设置可独立应用于标准计费和阶梯计费模式
- 用户开台前在页面上明确展示最低消费要求
- 系统自动在计费过程中应用最低消费规则

#### 4.3.4 特殊情况处理
- 若用户使用时长少于最低消费时长，系统自动按最低消费计算
- 开台确认页面显示最低消费提示（如"本桌最低消费30分钟"）
- 费用预览时优先考虑最低消费设置

计费过程实时展示功能在用户使用球桌期间，清晰地展示当前累计时长、实时费用和计费规则，提升使用透明度和用户体验。

#### 4.7.2 用户端展示设计
- **计时区域**：
  - 大号显示当前累计时长（时:分:秒）
  - 显示开始时间和预估结束时间
- **费用区域**：
  - 实时更新当前累计费用
  - 费用变化时有轻微动效提示
  - 显示当前所处计费阶段（阶梯计费时）

#### 4.7.3 计费规则展示
- 简洁展示当前适用的计费规则
- 阶梯计费时显示当前所处阶段和下一阶段信息
- 提供展开/收起详细计费规则的选项

#### 4.7.4 技术实现方案
- 前端计时器定时更新显示（每秒更新）
- 前端根据计费规则计算实时费用
- 定期（如每分钟）与服务端同步，确保计费准确性
- 使用动画和过渡效果优化用户体验

#### 4.7.5 显示优化
- 深色模式设计，减少屏幕亮度对球员的干扰
- 大字号设计，便于远距离查看
- 简洁明了的界面，减少干扰元素

  - 热门桌台排行：展示使用频率最高的桌台
  - 门店营收排行：展示不同门店的营收排行（多门店场景）

### 10.8 业务流程设计

2. 扣减会员积分
3. 创建积分消费记录
4. 关联业务场景和规则

2. 获取所有会员等级配置
3. 根据消费金额判断适用等级
4. 如果等级发生变化则更新会员信息
5. 设置新等级有效期
6. 可选：记录等级变更历史
7. 可选：发送会员升级通知

### 10.9 注意事项

1. 金额计算
   - 数据库中金额统一使用分为单位
   - 折扣计算结果保留2位小数
   - 设置合理的金额上下限

2. 积分规则
   - 合理设置积分有效期
   - 控制单次积分上限
   - 注意并发情况下的积分计算
   - 积分变动必须有明确的来源和说明

3. 等级管理
   - 设置合理的等级晋升门槛
   - 注意等级有效期的管理
   - 考虑等级降级的场景
   - 权益的有效期和等级有效期的关系

4. 数据安全
   - 积分和金额相关操作需要加事务
   - 关键操作需要记录操作日志
   - 敏感数据需要进行加密处理
   - 接口需要进行权限控制

5. 性能优化
   - 合理使用缓存
   - 分页查询大量数据
   - 定期清理过期数据
   - 异步处理非实时任务
